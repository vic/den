<Aside>
This guide shows how to create a Den config **from scratch**.

It is intended for people using a `configuration.nix` file as
generated by the NixOS installer and willing to adopt Den with it.

There's not much manual work to do here, but there's a lot of
insight on what is happening and Den concepts, so grab something
to drink while you read.

This guide assumes you have familiarity with:
- [`configuration.nix`](https://nlewo.github.io/nixos-manual-sphinx/configuration/configuration.xml.html)
- **No** Nix flakes. NixOS installer creates /etc/nixos/configuration.nix because that is the recommended, stable Nix way. This guide follows same path of not demanding flakes. See `From Flake to Den` guide if you use Nix unstable features like flakes.
- [`nixos-rebuild`](https://nixos.wiki/wiki/Nixos-rebuild) tool to build your OS 
- [`npins`](https://github.com/andir/npins) tool for fetching Den and dependencies.
</Aside>


### Step 1. A `Den`-dritic layout

Our first step is creating a directory layout for your Den.

```console "_nixos"
mkdir ./modules                     # Dendritic modules directory
cp -r /etc/nixos ./modules/_nixos  # OPTIONAL existing NixOS modules
```

<Steps>
1. All your **Dendritic Nix modules** will be loaded by [`import-tree`](https://github.com/vic/import-tree) automatically from `./modules`
2. _(optional)_ Copy your existing _NixOS modules_ into `modules/_nixos`. 

   The `_` is **important** for import-tree to **ignore** and _NOT_ auto-load `_nixos/configuration.nix` nor `_nixos/hardware-configuration.nix` since they are regular NixOS modules and not Dendritic ones.
</Steps>

### Step 2. Pinning Den and dependencies

This step will use `npins` to fetch and lock some dependencies that
will integrate all together:

```console "import-tree" "flake-aspects" "den" "with-inputs"
npins init                                  # adds nixpkgs channel
npins add github vic import-tree   -b main  # for auto-importing ./modules
npins add github vic flake-aspects -b main  # aspects used by Den
npins add github vic den           -b main  # Den itself, of course
npins add github vic with-inputs   -b main  # flake-like inputs without Nix flakes.
npins add github nix-community home-manager --branch master # OPTIONAL home integration
```

### Step 3. Your Den Entry Point

Lets create `default.nix`. This file will serve as the entrypoint to all your configuration. Once you have `default.nix` you will **very rarely** need to touch it again.

```nix "import ./npins" "with-inputs sources { }" "lib.evalModules" "import-tree ./modules" "specialArgs.inputs" "config.flake" "with-inputs outputs"
# default.nix
let
  sources = import ./npins; # (1)
  with-inputs = import sources.with-inputs sources { }; # (2)
  outputs = inputs:
    (inputs.nixpkgs.lib.evalModules { # (3)
      modules = [ (inputs.import-tree ./modules) ]; # (4)
      specialArgs.inputs = inputs; # (5)
    }).config.flake; # (6)
in
with-inputs outputs # (7)
```

A tiny file, but a lof of things happening on it:

<Steps>
1. Import fetched and already locked dependencies using `npins`.

2. Flake like input follows and overrides via [with-inputs](https://github.com/vic/with-inputs). 

    This ensures full compatibility between flakes and non-flakes worlds. You can share and re-use Dendritic modules from other people independently if they use flakes or not. Also in case you later move into flakes, nothing in your modules changes, just the entrypoint.

3. Merges all imported modules into an actual unified config.

4. Single usage of import-tree to load all modules recursively.

5. Provide `{ inputs, ... }` special argument to all modules.

6. Read from Den's `flake` output where `nixosConfigurations` are placed for compatibility between flakes and non-flakes environments.

7. Invokes `output` with resolved `inputs`.

</Steps>

### Step 4. Host configuration in Den

<Aside title="No Monoliths" icon="nix">
For brevity and teaching purposes, this guide uses a single `modules/den.nix` file. But Dendritic setups are not monolithic unlike most flake examples in the wild.

The recommendation is to split your modules as they make sense to you. Any file can contribute to any Den aspects and hosts/users can be defined anywhere.

Recommended read: [Dendritic Advantages](https://dendrix.oeiuwq.com/Dendritic.html#dendritic-advantages) and [Dendritic FAQ](https://github.com/Doc-Steve/dendritic-design-with-flake-parts/wiki/FAQ)
</Aside>

```nix "x86_64-linux.igloo" "users.tux" "den.flakeModule" "aspects.igloo" "aspects.tux" "user.classes" "_nixos/configuration.nix" "hostname" "includes =" "define-user" "primary-user" "(1)"  "(2)"  "(3)"  "(4)"  "(5)"  "(6)"  "(7)"  "(8)"  "(9)"
# modules/den.nix
{ inputs, den, lib, ... }: {
  imports = [ inputs.den.flakeModule ]; # (1)

  den.schema.user.classes = lib.mkDefault [ "homeManager" ]; # (2)

  den.default.homeManager.home.stateVersion = "25.11"; # (3)

  den.hosts.x86_64-linux.igloo.users.tux = {}; # (4) (5)

  den.aspects.igloo = { # (6)
    includes = [ den.provides.hostname ]; # (7)
    nixos = { pkgs, ... }: {
      imports = [ ./_nixos/configuration.nix ]; # (8)
      environment.systemPackages = [ pkgs.hello ];
    };
  };

  den.aspects.tux = { # (9)
    includes = [ den.provides.define-user den.provides.primary-user ]; # (10)
    homeManager = { pkgs, ... }: {
      home.packages = [ pkgs.vim ];
    };
  };
}
```

This is all that is happening at `modules/den.nix`.

<Steps>

1. Imports `den`-framework features into the module system. 

    This provides the `den` module argument as well as all `den.*` options, `nixosConfigurations` outputs.

2. _(optional, if you added home-manager dependency)_. This example shows how to enable `homeManager` integration for all users by default. 

    Den supports multiple home environments, that is why none of them is enabled by default in Den. Freedom of choice is important, same as with flakes.

    Home Integration is a User concern, so it is configured at each independent `<user>.classes` or `den.schema.user` for all of them.
    See [User Declaration](/guides/declare-hosts#user-declaration) and [Homes Integration](/guides/home-manager) for more.

    <Aside icon="nix" title="Nix Classes">

    An flake-aspect (used by Den) is just an attrset that contains modules
    of different Nix `class`.

    ```nix
    gaming = { nixos = ...; homeManager = ...; hjem = ...; }
    ```

    Such an attrset is said to be **Dendritic** because it allows configuring a single cross-cutting concern (`gaming`) over different Nix classes.

    Nix classes are NOT an artificial concept created by Den. There are [many nix classes in the wild](https://github.com/search?q=language%3ANix+%28%22+class+%3D+%22+AND+%22evalModules%22%29&type=code), the most common are [`nixos`](https://github.com/search?q=repo%3ANixOS%2Fnixpkgs+%22class+%3D+%5C%22nixos%5C%22%22&type=code), [`darwin`](https://github.com/nix-darwin/nix-darwin/blob/da529ac9e46f25ed5616fd634079a5f3c579135f/eval-config.nix#L81), [`homeManager`](https://github.com/nix-community/home-manager/blob/5be5d8245cbc7bc0c09fbb5f38f23f223c543f85/nixos/common.nix#L27).
    </Aside>

3. **den.default** applies settings to **all** hosts, users, and homes. This is the right place for `stateVersion` and other global policies.

4. **Host definitions** go inside `den.hosts`.

    This single line defines a **host** with hostName `igloo`, platform `x86_64-linux` from which `class = "nixos"` is derived, and a single user `tux` in it. See [Declare Hosts & Users](/guide/declare-hosts) for more.

    Den **automatically** creates an aspect for each host that looks like this:

    ```nix "igloo" "parametric"
    den.aspects.igloo = parametric {
      nixos = {};
    };
    ```

    `parametric` means that this aspect is able to pass its context (the `{host}` definition for `igloo` into other aspects being included in it.
    See [Parametric Aspects](/explanation/parametric) for more this.

5. **User definitions** inside host.

    Hosts can have zero or more users. This example shows only one: `tux`. 

    Each user can define its own configuration via freeform attributes, or define values for options from `den.schema.user`, like `classes`. See [Schema Reference](/reference/schema) for more.

    ```nix "homeManager" "hjem" "maid"
    den.hosts.x86_64-linux.igloo.users = {
      tux = { 
        likes = [ "fishes"; ];
        classes = [ "homeManager" "hjem" ];
      };
      pingu = {
        likes = [ "music" ];
        classes = [ "maid" ];
      };
    };
    ```

    Just like hosts, users also get an aspect **automatically** created by Den from their definition:

    ```nix "homeManager" "hjem" "maid"
    den.aspects.tux = parametric {
      homeManager = {};
      hjem = {};
    };

    den.aspects.pingu = parametric {
      maid = {};
    };
    ```

6. **Enhance** the `igloo` host aspect.

    Remember that `den.aspects.igloo` was _already_ created for you by Den from the host definition?. Well, in this step **you are extending** the aspect with more configuration. Any file can do this, contribute to any host or user aspect as they please. This is called [**Incremental Features**](https://dendrix.oeiuwq.com/Dendritic.html#incremental-features) in the Dendritic Advantages article.

7. Include re-usable features from other, generic aspects.

    One of the advantages of [`flake-aspects`](https://github.com/vic/flake-aspects)' upon which Den is built is automatic inclusion of other aspects. Unlike NixOS `.imports`, `.includes` are safe to use with conditionals because includes are not files to be imported.

    Den exploits flake-aspects by using contexts. A context like `{host}` is simply an attribute set that holds the `igloo` host definition from our examples. These are **arguments** to real Nix functions, NOT `_module.args` nor `specialArgs`. This is why these values can be used for conditional includes.

    Den **provides** some common batteries. Re-usable aspects like
    [`den.provides.hostname`](https://github.com/vic/den/blob/main/modules/aspects/provides/hostname.nix) which is defined like this:

    ```nix "host," "host.hostName"
    den.provides.hostname = { host, ... }: { # host = igloo definition
      nixos.networking.hostName = host.hostName; # value from Host Schema
    };
    ```

8. _optional if you copied `/etc/nixos` into `modules/_nixos`)_ Import Non-Dendritic **NixOS modules**.

    The `nixos` attribute value is just a regular NixOS module. Because of this, it can import your previous `./_nixos/configuration.nix` module or any other NixOS module from Nix libraries.


9. Enhance the User aspect.

    As with the host aspect, Den also **automatically** created a user aspect. Here, you are extending it with more configuration.

10. Include re-usable `{host, user}` batteries.

    Batteries like [`den.provides.primary-user`](https://github.com/vic/den/blob/main/modules/aspects/provides/primary-user.nix) take the `{host, user}` context where `host` is the `igloo` definition and `user` is the `tux` definition.

    These batteries also serve as example for you to create custom re-usable pieces of configuration shareable on several user/hosts or shared with the community.

</Steps>
